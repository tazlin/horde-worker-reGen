# Utilities venv dependency lock

This directory holds the dependency definition for the worker's **second virtual environment**, the
image-utilities capability venv. That venv runs the `horde-image-utilities` capability service in an
environment separate from the worker's own `.venv`, so its native, accelerator-gated dependencies never
share a resolution with the worker's dependencies.

## Layout

```
requirements/utilities/
  pyproject.toml   # the utilities service + torch, one extra per locked build
  uv.lock          # the committed, fully-resolved lock (generated by `uv lock`)
```

`requirements/utilities/` is a self-contained uv project. The bootstrap provisions the utilities venv from
it with `uv sync --locked --extra <backend_token>` (pointing `UV_PROJECT_ENVIRONMENT` at the peered
utilities venv). `<backend_token>` is one of the locked torch-build tokens the bootstrap resolves (`cu126`,
`cu130`, `cu132`, `cpu`).

## Why a lock, not a floating requirements file

A locked sync installs strictly from the lockfile's **recorded sources** and never re-resolves. That is the
whole point:

- **Deterministic.** Every machine and every run gets byte-identical wheels.
- **Cache-shared.** The lock pins `torch`/`torchvision` to the *same* build the root worker's `uv.lock`
  resolves, so the utilities venv hardlinks the main `.venv`'s already-cached torch/CUDA wheels from the
  peered uv cache instead of re-downloading multi-GB of torch.
- **Immune to ambient indexes.** A floating `uv pip install -r requirements.txt` re-resolves against
  whatever indexes are configured on the box (an ambient `UV_EXTRA_INDEX_URL`, a `pypi.nvidia.com` in
  `~/.config/uv/uv.toml`, ...) and can drift onto a different, sometimes unreachable, source. A locked sync
  cannot: it installs the exact wheels the lock names, exactly as the root worker's own locked sync does.

## torch/torchvision pinning

`requirements/utilities/pyproject.toml` pins `torch`/`torchvision` (via `[tool.uv]
constraint-dependencies`) to the same version the **root** `uv.lock` resolves, and routes the build
(`+cuXXX`/`+cpu`) through the same `[tool.uv.sources]` + `[[tool.uv.index]]` PyTorch indexes the root project
uses. The root worker leaves torch unpinned but is capped lower by horde-engine's torchvision range; the
service has no such cap and would otherwise float ahead. `tests/bootstrap/test_utilities_env.py` fails if the
two locks disagree on the torch/torchvision build, so a torch bump in the root lock is a deliberate re-pin +
re-lock here (see below).

## Regenerating the lock

After changing `requirements/utilities/pyproject.toml` (a `horde-image-utilities` bump, a torch re-pin, an
extra change), regenerate the lock in a clean environment:

```bash
cd requirements/utilities
uv lock
```

CI asserts the committed lock is current with `uv lock --check`, so a stale lock fails the build. Do **not**
add `mediapipe` to the annotator extras: the current service release declares an OpenCV wheel conflict
between those extras.

## Provisioning

The bootstrap (`worker_bootstrap`) provisions the utilities venv after a successful worker sync, and also
repairs it on a launch whose main venv is already current. It acts when the resolved feature set is non-empty
and the utilities lock is present. See `docs/explanation/compute_backends.md` and
`docs/explanation/image_utilities_lane.md` for the full flow and the `HORDE_WORKER_FEATURES` /
`HORDE_WORKER_SKIP_UTILITIES` controls.
